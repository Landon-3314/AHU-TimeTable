# M4b 时间变化、诊断结果与损坏行隔离设计

## 背景

`problem.md` 中仍有四个相互关联的可靠性缺口：

1. Android 原生闹钟只保存绝对时间戳。系统时区、系统时间或日期变化后，已注册计划不会按新的本地时间语义重排。
2. 开发者诊断页调用原生静音测试时，MethodChannel 无论闹钟是否真正写入成功都返回成功，可能产生误导。
3. `Event.fromJson` 在日程时间损坏时使用当前时间兜底，会把损坏数据伪装成新的有效日程。
4. 存储列表读取会静默跳过损坏行。后续保存可能覆盖原始坏行，用户既看不到提示，也无法保留诊断副本。

本里程碑将四项一起修复，保持结构性损坏严格恢复，同时允许单行损坏在隔离后降级启动。

## 目标

- Android 计划在时区、系统时间或日期变化后重新注册。
- 时区变化后，课程与日程保持原本的本地日期时间语义，并正确处理夏令时。
- 原生静音诊断只有在静音与恢复闹钟均写入成功时才报告成功。
- 非法日程时间不再伪造成当前时间。
- 单行损坏在被覆盖前保存本地诊断副本。
- 若外部备份不可恢复，则清理已隔离坏行并保留其余有效数据继续启动。
- 首页对新隔离记录显示一次性提示。

## 非目标

- 不增加面向普通用户的诊断记录浏览器或导出入口。
- 不修改外部业务备份 schema。
- 不尝试猜测或修复损坏行字段。
- 不对结构性损坏做部分修复，包括损坏的学期列表、学期引用、迁移元数据和操作日志。
- 不在本里程碑处理课程冲突、考试独立视图、撤销、搜索筛选、深色模式或国际化等其余 P2/P3 条目。

## Android 时间变化重排

### 持久化

`NativeStateStore` 随原生闹钟项目保存计划生成时的时区 ID。旧版本升级后若没有时区 ID，首次重排视为当前时区，避免在缺少来源信息时盲目移动时间戳。

保存闹钟项目时同步保存当前时区 ID；清空闹钟项目时一并删除该 ID。

### 广播监听

`BootRescheduleReceiver` 在现有启动、应用升级和精确闹钟权限变化广播之外，监听：

- `Intent.ACTION_TIMEZONE_CHANGED`
- `Intent.ACTION_TIME_CHANGED`
- `Intent.ACTION_DATE_CHANGED`

以上广播均重新调用原生计划重排和前台状态刷新。

### 时区重建规则

新增纯策略模块 `NativeAlarmTimePolicy`。当已保存时区与当前时区不同，策略将每个非空时间戳按以下方式重建：

1. 使用来源时区读取原时间戳的年、月、日、时、分、秒和毫秒。
2. 使用目标时区和相同本地时间字段生成新时间戳。
3. 将 `AlarmItem` 中所有时间字段统一重建，包括静音、恢复、提醒、状态窗口开始和状态窗口结束。
4. 将前台状态展示使用的今日课程缓存开始和结束时间一并重建。
5. 重建后再过滤没有未来工作的项目，并按现有逻辑重新注册。

实现使用 Android 兼容的 `java.util.Calendar` 与 `java.util.TimeZone`，避免引入额外 API 级别要求。该规则支持 DST，不使用简单 UTC 偏移量加减。

## 诊断静音真实结果

### 原生结果模型

原生调度器返回结构化结果：

```text
DiagnosticScheduleResult(
  success: Boolean,
  reason: String?
)
```

只有静音闹钟和恢复闹钟都成功注册时 `success` 才为 `true`。失败原因至少包含：

- `silent_alarm_schedule_failed`
- `restore_alarm_schedule_failed`

恢复闹钟允许沿用现有非精确降级路径；若降级写入也失败，才报告恢复失败。静音闹钟不能在写入失败后继续报告成功。

### MethodChannel 与 Dart

`MainActivity` 将原生结果作为 `{success, reason}` 返回。`NativeAlarmService` 解析为 Dart 结果对象，并对平台异常返回通用失败结果。

开发者诊断页保留现有按钮与 SnackBar 交互：

- 成功：显示已写入测试闹钟。
- 静音写入失败：显示静音闹钟写入失败。
- 恢复写入失败：显示恢复闹钟写入失败。
- 未知失败：显示通用失败提示并引导查看 `MuteDiag` 日志。

## 损坏行隔离

### 严格模型解析

`Event.fromJson` 要求 `dateTime` 为可解析字符串。字段缺失、类型错误或格式错误时抛出 `FormatException`，不再使用 `DateTime.now()` 兜底。

### 单行与结构性损坏区分

存储层增加行级扫描：

- 可解析列表中若只有个别课程或日程 JSON 行损坏，视为行级损坏。
- JSON 列表类型错误、学期列表损坏、学期 ID 关系错误、迁移元数据错误或操作日志损坏，仍视为结构性损坏。

结构性损坏沿用现有严格恢复路径，不尝试清理后继续。

### 本地诊断区

新增独立本地偏好键保存最近 100 条隔离记录。每条记录包含：

```text
sourceKey
rawValue
reason
detectedAt
```

另设待提示计数键。诊断区键不以 `semesters.`、`settings.`、`onboarding.` 或 `updates.` 开头，因此不会进入外部业务备份。

隔离去重键为 `sourceKey + rawValue`。相同坏行不会重复写入诊断区，也不会重复增加待提示数量。超过 100 条时删除最旧记录。

### 启动恢复顺序

`StorageService.create` 按以下顺序处理：

1. 扫描内部课程与日程列表，收集并持久化行级损坏诊断副本。
2. 分类内部状态。
3. 若内部数据有效，继续启动。
4. 若内部数据损坏，优先尝试外部备份恢复。
5. 若外部备份恢复成功，保留诊断副本并使用恢复后的有效业务数据。
6. 若无可用外部备份，且损坏仅限已隔离的行，则删除坏行并保留有效行继续启动。
7. 若仍存在结构性损坏，则保持初始化失败，不做猜测性修复。
8. 仅在业务数据重新分类为有效后同步外部备份。

该顺序确保原始坏行在任何恢复、清理或后续用户保存之前被保留。

## 一次性首页提示

`StorageService` 提供消费待提示数量的方法。应用初始化成功后将数量放入初始化结果，首页首次构建后显示一次 SnackBar：

```text
已跳过并保留 N 条损坏日程记录
```

读取时同时清除待提示计数，因此同一批记录只提示一次。提示涵盖课程和日程损坏行，文案使用“日程记录”作为用户可理解的总称。

## 测试策略

### Kotlin JVM 单测

- 同一时间戳在不同时区之间保持本地时间字段。
- 跨 DST 时使用目标时区规则重建，不做固定偏移量移动。
- 缺少已保存时区时不盲目移动时间戳。
- 诊断结果仅在静音与恢复均成功时成功。
- 诊断结果区分静音失败与恢复失败。
- 广播接收器白名单包含时区、系统时间与日期变化广播。

### Dart 单测

- 非法 `Event.dateTime` 抛出 `FormatException`。
- 损坏课程或日程行在启动时写入诊断区。
- 相同坏行重复扫描不会重复记录或增加待提示数量。
- 诊断区最多保留最近 100 条。
- 外部备份可用时优先恢复，并保留诊断记录。
- 外部备份不可用时删除坏行并保留有效行启动。
- 结构性损坏在无备份时仍初始化失败。
- 待提示数量消费后归零。
- MethodChannel 结构化结果可解析，异常映射为通用失败。
- 开发者诊断页显示对应中文失败提示。

## 风险控制

- 时区重建仅在来源时区明确且与当前时区不同时发生。
- 诊断区排除在外部业务备份之外，避免损坏行污染恢复快照。
- 行级降级仅处理课程与日程字符串列表中的个别坏行。
- 外部备份仍优先于本地清理，最大化恢复完整数据的机会。
- 所有新增行为按 TDD 顺序实施，先观察失败测试，再写最小修复。
