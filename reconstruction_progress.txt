一、核心依赖与架构现状
==================================================

1. 关键三方包与当前作用域

- provider
  作用域：全局状态管理。
  当前落点：
  - lib/main.dart 通过 MultiProvider 注入 SettingsProvider / CourseProvider。
  - UI 通过 context.watch / read 直接消费 Provider。
  现状判断：
  - Provider 已承担“状态 + 持久化 + 调度触发”三类职责，边界偏厚。

- shared_preferences
  作用域：本地持久化。
  当前落点：
  - CourseProvider 持久化课程和日程：`courses.items`、`events.items`
  - SettingsProvider 持久化课表参数、提醒参数、语言、学期信息、自动静音开关
  - background_service.dart 在后台 isolate 中直接读取同一批 key
  现状判断：
  - 当前是单一事实来源，但 key 分散在多个文件中重复定义，存在耦合风险。

- webview_flutter
  作用域：教务系统登录与页面 DOM 抓取。
  当前落点：
  - lib/screens/import_course_page.dart
  - lib/screens/settings_page.dart 中用于清理 WebView Cookie
  现状判断：
  - 导入链路完全依赖页面结构和 JS 注入，核心解析逻辑目前全部堆在 UI 页面文件中。

- intl
  作用域：日期时间格式化。
  当前落点：
  - timetable_page.dart
  - settings_page.dart

- flutter_local_notifications
  作用域：Android/iOS 通知初始化与即时提醒展示。
  当前落点：
  - lib/services/notification_service.dart：创建通道、申请权限、保留历史兼容 API
  - lib/background_service.dart：后台轮询命中后即时 `show(...)`
  现状判断：
  - 真正生效的课程提醒发送端在 background_service.dart。
  - NotificationService 当前已退化为“初始化 + 通道管理 + 兼容壳”。

- flutter_background_service
  作用域：Android 前台常驻后台服务。
  当前落点：
  - lib/background_service.dart
  - lib/main.dart 通过 `requestBackgroundServiceSync()` / `stopBackgroundServiceIfRunning()` 间接驱动
  现状判断：
  - 当前自动静音与课程提醒共用一个后台轮询服务。
  - 服务编排、静音计算、提醒计算、去重、通道配置均混在单文件。

- sound_mode
  作用域：Android 铃声模式切换与权限查询。
  当前落点：
  - lib/background_service.dart：后台实际执行 `SoundMode.setSoundMode(...)`
  - lib/services/auto_mute_service.dart：方法通道能力封装与权限检查
  现状判断：
  - 真正生效的静音切换在 background_service.dart。
  - AutoMuteService 当前主要只剩“权限检查 / 打开授权页”仍在被 SettingsProvider 调用。

- app_settings
  作用域：跳转系统电池优化设置。
  当前落点：
  - lib/screens/settings_page.dart

- timezone
  作用域：依赖已声明，但当前 lib/ 无实际引用。
  现状判断：
  - 属于历史遗留依赖，可在后续重构中复核是否保留。

2. 当前项目有效分层

- 启动层
  - lib/main.dart
  - 负责初始化 SharedPreferences、NotificationService、Provider 注入、提醒刷新绑定。

  当前特征：
  - 服务职责没有彻底隔离。
  - background_service.dart 实际上已兼具配置中心、调度器、静音引擎、提醒引擎四类职责。

- 视图层
  - lib/screens/timetable_page.dart
  - lib/screens/add_course_page.dart
  - lib/screens/import_course_page.dart
  - lib/screens/settings_page.dart

  当前特征：
  - timetable_page.dart 和 import_course_page.dart 明显混入较重业务逻辑。

- 模型层
  - lib/models/course.dart
  - lib/models/event.dart

  当前特征：
  - Course 无稳定 id，增删改依赖结构相等或对象引用。
  - Event 有 id，但提醒消费链未完全接通。

- 文案层
  - lib/app_localizations.dart
  当前特征：
  - 简单字典式国际化，未引入标准 l10n 生成链路。

3. 当前真实运行主链路

- 应用启动
  - main() -> SharedPreferences.getInstance()
  - main() -> NotificationService.instance.initialize()
  - MainApp.initState() -> 构造 SettingsProvider / CourseProvider / TimetableViewProvider
  - MainApp.initState() -> bindReminderScheduler(refreshReminders)
  - MainApp.initState() -> TimetableViewProvider.initializeRealDate(...)
  - MainApp.initState() -> unawaited(refreshReminders())

- 提醒刷新主入口
  - refreshReminders() 定义在 lib/main.dart
  - 会并行执行：
    - NotificationService.instance.refreshAllReminders(...)
    - 自动静音开启时 `requestBackgroundServiceSync()`
    - 自动静音关闭时 `stopBackgroundServiceIfRunning()`
  现状判断：
  - NotificationService.refreshAllReminders 当前仅初始化并打印日志，不做真正调度。
  - 真正的调度入口已经转移到 background_service。

- 数据事实来源
  - 前台 Provider 与后台 service 都直接基于 SharedPreferences 同步状态。
  - 这使得“无需数据库即可共享状态”跑通了闭环，但也把 key、序列化规则和业务判断散落到了多个文件中。

4. 当前明确存在的遗留/兼容层

- lib/services/notification_service.dart
  - `scheduleAllCourseReminders`、`scheduleEventReminders`、`refreshAllReminders`
  - 当前均为兼容壳，真实逻辑已迁移。

- lib/services/auto_mute_service.dart
  - `scheduleMuteTasks(...)` 目前在 lib/ 中没有任何调用点。
  - 方法通道 `replaceTasks` / `setSilentNow` 为历史方案残留接口。

- lib/background_service.dart
  - `initBackgroundService()` 目前在 lib/ 中没有直接调用点。
  - 实际配置动作由 `requestBackgroundServiceSync()` 间接触发。

- android_alarm_manager_plus
  - 当前 `pubspec.yaml` 未声明该依赖。
  - 也就是说，现阶段有效代码路径并不依赖它；若后续要重构后台能力，应以“当前真实运行链路”为准，而不是以历史描述为准。

A. 启动与全局装配

1. Provider 装配闭环
- 入口：lib/main.dart
- 过程：
  - 初始化 SharedPreferences
  - 初始化 NotificationService
  - 构造 SettingsProvider / CourseProvider / TimetableViewProvider
  - 将 `refreshReminders` 回调绑定给 SettingsProvider / CourseProvider
  - 用 SettingsProvider 的 `currentRealWeek/currentRealWeekday` 初始化 TimetableViewProvider
- 当前效果：
  - 启动后 UI 会定位到“真实今天所在周/星期”
  - 数据和设置变更后可统一触发后台同步入口

2. 提醒刷新统一出入口
- 入口：
  - CourseProvider 中所有增删改课程/日程操作
  - SettingsProvider 中所有影响课表时间、学期周次、提醒、自动静音的设置更新
- 过程：
  - Provider 完成本地持久化后调用 `_refreshReminders()`
  - `_refreshReminders()` 最终回到 main.dart 中的 `refreshReminders()`
- 当前效果：
  - 所有“会影响提醒/静音”的数据改动都会推动后台服务重新同步

C. 课表时间轴与学期配置闭环

1. SettingsProvider 的课表参数模型
- 文件：lib/providers/settings_provider.dart
- 持久化字段：
  - 时间轴：pixelsPerMinute
  - 节次规则：classDuration / shortBreak / bigBreak
  - 三段作息：morning/afternoon/evening 的开始时间与节次数
  - 学期：semesterStartDate / totalWeeks
  - 提醒：reminderAdvanceMinutes / eventReminderAdvanceMinutes
  - 其他：languageCode / autoMuteEnabled

2. 真实周次与日期换算
- 核心 API：
  - `currentRealWeek`
  - `currentRealWeekday`
  - `getDateFor(week, weekday)`
- 核心逻辑：
  - 学期开始日期在写入时会被 `_alignToMonday(...)` 强制对齐到周一
  - 当前周次用 `today.difference(_semesterStartDate).inDays ~/ 7 + 1` 计算
- 当前效果：
  - 课表页、假期判断、后台静音/提醒均依赖同一套周次计算

3. TimeSlot 生成闭环
- 核心 API：
  - `SettingsProvider.generateTimeSlots()`
  - `SettingsProvider.timeSlots`
- 核心逻辑：
  - 分早/午/晚三段生成节次
  - 每段第 2 节后插入大课间，其余节间插入小课间
- 当前效果：
  - UI 展示时间、课程详情时间、后台判断上课中、课程提醒触发时间，都依赖这套节次生成结果


D. TimetablePage 多视图渲染与状态机闭环

1. 页面模式
- 文件：lib/screens/timetable_page.dart
- 模式：
  - `_TimetableMode.day`
  - `_TimetableMode.week`
- 当前效果：
  - 顶部 SegmentedButton 可切换日视图 / 周视图

2. 双 PageController 联动
- 控制器：
  - `_dayPageController`
  - `_weekPageController`
- 同步状态：
  - `_selectedWeekForWeekView`
  - `TimetableViewProvider.currentWeek/currentWeekday`
  - `_isSyncingControllers`
- 核心流转：
  - 日视图翻页 -> 计算绝对日 index -> 推导周/星期 -> 回写 Provider -> 同步周视图页码
  - 周视图翻页 -> 回写周选择状态 -> 同步日视图绝对 index
  - “今天”按钮 -> 同步跳转两个控制器
  - 周跳转菜单 -> 支持普通周和“假期中”
- 当前效果：
  - 日/周双视图与顶部周选择器可保持联动
  - 这是一个真实存在的页面级状态机，后续需要单独拆出

3. 日视图渲染闭环
- 入口：DayViewWidget
- 数据流：
  - 从 CourseProvider 拉取 courses / events
  - 依据 `(week, weekday)` 过滤当日课程与当日日程
  - 课程排序依据 startPeriod
  - 日程排序依据 event.dateTime
  - 最终合成为 `_DayScheduleItem`，按实际发生时间再做一次总排序
- 关键点：
  - 课程排序时间通过 `_courseStartDateTime(...)` 转成真实 DateTime
  - 因而实现了“课程 + 日程混合时间线”
- 当前效果：
  - 同一天内课程和事件可混排显示

4. 周视图渲染闭环
- 入口：WeekViewWidget
- 数据流：
  - 普通周：按周过滤课程与日程，再按 weekday 分组
  - 假期页：使用 `itemCount = totalWeeks + 1`，最后一页为 Holiday
  - Holiday 过滤规则：事件时间早于学期第一周开始，或晚于学期最后一周结束
- 当前效果：
  - 普通周按 7 天分段显示课程和事件
  - 最后一页固定承载“假期中”事件列表

5. 课程详情 / 事件详情 / 删除 / 编辑
- 入口：
  - `showCourseDetailsSheet(...)`
  - `showEventDetailsSheet(...)`
- 课程详情特点：
  - 会读取 SettingsProvider.timeSlots 计算课程实际起止时间
  - 编辑会跳转 AddCoursePage(existingCourse: course)
  - 删除直接调用 CourseProvider.removeCourse(course)
- 事件详情特点：
  - 展示 enableAlarm 状态，但当前仅是显示字段
  - 删除调用 CourseProvider.deleteEvent(event.id)

G. 后台自动静音与课程提醒闭环

1. 后台服务配置入口
- 文件：lib/background_service.dart
- 核心 API：
  - `requestBackgroundServiceSync()`
  - `stopBackgroundServiceIfRunning()`
  - `onServiceStart(ServiceInstance service)`
- 配置流程：
  - 创建 `silent_bg_channel` 与 `course_reminder_channel`
  - `FlutterBackgroundService.configure(...)`
  - Android 使用前台服务模式，通知 id 固定为 9527

2. 服务启动后的行为
- `onServiceStart(...)` 中会：
  - `WidgetsFlutterBinding.ensureInitialized()`
  - `DartPluginRegistrant.ensureInitialized()`
  - 初始化 FlutterLocalNotificationsPlugin
  - 监听 `sync_now` / `stop_service`
  - 启动 `Timer.periodic(Duration(minutes: 1))`
  - 首次立即执行一次 `runScheduleTick()`

3. 自动静音当前如何跑通
- 核心链路：
  - SettingsPage 切换开关
  - `SettingsProvider.toggleAutoMuteWithCheck(...)`
  - Android 下通过 `AutoMuteService.instance.hasPermission()` 检查免打扰权限
  - 开启后走 `refreshReminders()` -> `requestBackgroundServiceSync()`
  - 后台每分钟 `runScheduleTick()` -> `_computeTargetModeFromPrefs(...)`
  - `_computeTargetModeFromPrefs(...)` 直接读 SharedPreferences 中的课程、学期、节次配置
  - 命中上课中时返回 `vibrate`，否则返回 `normal`
  - `applyMode(...)` 调用 `SoundMode.setSoundMode(...)`
- 当前效果：
  - 自动静音已经不依赖前台页面常驻
  - 只要后台服务存活，就能按当前课程表切换铃声模式

4. 课程提醒当前如何跑通
- 核心链路：
  - 后台每分钟 `runScheduleTick()`
  - `_runCourseReminderTick(...)`
  - 读取 `settings.reminderAdvanceMinutes`
  - 读取当前周次、星期与课程列表
  - 计算课程开始时间 `classStart`
  - `reminderTime = classStart - advanceMinutes`
  - 通过 `_isSameMinute(now, reminderTime)` 命中当前分钟
  - 使用 `FlutterLocalNotificationsPlugin.show(...)` 即时发送通知
  - `dedupeId = '$dayKey-$currentWeek-$name-$startPeriod'`
  - `notifiedCourseIds` 按天重置，防止同一分钟重复通知
- 当前效果：
  - 课程提醒已从“未来定时调度”迁移为“后台分钟级轮询 + 即时 show”

5. 当前提醒链路的重要真实状态
- NotificationService 只是兼容层，不负责任何未来调度
- 课程提醒只处理 Course，不处理 Event
- `eventReminderAdvanceMinutes` 目前只存在于设置存储与 UI 文案，不参与后台轮询判断
- `Event.enableAlarm` 目前只会在表单保存和详情页展示，不参与提醒命中

  - 主界面、设置、导入、增改页已使用统一取词入口

2. 当前问题
- 国际化仍是单文件硬编码字典
- 页面中仍夹杂少量直接拼装文案和语言判断分支


J. 本轮建档确认出的关键风险点

- Risk 1：课表导航状态机散落在 TimetablePage 内部
  - 周选择、绝对日 index、PageController、Provider currentWeek/currentWeekday 相互缠绕

- Risk 2：后台服务对 SharedPreferences key 与序列化结构高度硬编码
  - 任何模型字段改名都会直接影响后台静音/提醒

- Risk 3：导入解析器完全与页面耦合
  - JS 脚本体量大，无法做最小单元测试

- Risk 4：课程无 stable id
  - 编辑、删除、详情存在性判断都存在后续误判风险

- Risk 5：Event 提醒是“UI 可配、数据可存、后台未消费”的半接通状态
  - 后续重构时不能误以为它已经完整跑通
