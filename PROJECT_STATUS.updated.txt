Overview
该项目当前已从“课程表展示”演进为“本地优先的课程+日程管理应用”，并完成以下核心方向：
- 课程（Course）与单次日程（Event）混合管理
- 教务系统 WebView 抓取导入（含复杂/异常结构兼容）
- 多语言（中文/英文）
- Android 自动静音（前台常驻后台服务）
- 课前提醒迁移至后台轮询触发（不再依赖 zonedSchedule）
- 周视图支持“假期中”模式

Current Architecture (Active)
- UI: PageView + ListView + Provider
- Local storage: SharedPreferences
- Import: WebView + JS 注入 DOM 抓取
- Background runtime: flutter_background_service (Android foreground service)
- Notifications: flutter_local_notifications (即时 show，不再做未来时刻注册)

High Coupling / Low Robustness Areas (Rewrite Backlog)
以下模块建议在“功能基本稳定后”优先重写：

[Priority P0]
A) Import JS Parser (lib/screens/import_course_page.dart -> extractScript)
- 问题：单文件内注入脚本体量过大，正则/DOM分支耦合高，回归风险大
- 表现：每次新增特例都在同一函数堆逻辑，维护成本持续上升
- 建议：
  - 将 JS parser 模块化（清洗、分段、字段抽取、周次解析、去重分层）
  - 引入可回放样本测试集（保存 HTML 夹具）做回归测试

B) Background Service Monolith (lib/background_service.dart)
- 问题：静音控制、提醒匹配、通知触发、去重状态、课表时间计算全部耦合在一个文件
- 风险：任何一处变更都可能影响多功能链路
- 建议：
  - 拆为 service coordinator + mute engine + reminder engine + schedule matcher
  - 将 SharedPreferences 键映射集中管理，避免散落常量

[Priority P1]
C) TimetablePage 状态耦合（lib/screens/timetable_page.dart）
- 问题：Day/Week/Holiday 的周状态同步逻辑与页面控制器耦合重
- 风险：切周、跳转今天、假期模式切换时容易出现边界状态冲突
- 建议：
  - 抽离 WeekSelectionState（含普通周/假期周统一状态机）
  - 将 WeekView 数据过滤逻辑下沉到 provider/service

D) NotificationService 兼容外壳
- 问题：当前已迁移后台轮询，但 NotificationService 仍保留历史 API 壳
- 风险：后续维护者误用旧接口导致行为混乱
- 建议：
  - 明确 deprecate 标记
  - 清理无效接口或改成纯 channel/permission manager

[Priority P2]
E) Course 身份唯一性
- 问题：暂无 stable id，多个结构相同课程可能被误判为同一条
- 影响：详情页安全校验、编辑定位、去重策略不够可靠
- 建议：为 Course 增加 stable id（导入、编辑、删除全链路改造）

F) 调试日志噪音
- 问题：Main/Provider/BackgroundService 日志较多，线上排障与业务日志混杂
- 建议：引入日志级别与开关（debug/info/warn/error）
